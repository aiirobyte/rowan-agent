主流程分支

Direct route

routeRequest() 返回 route: "direct" 后，loop 调 createDirectOutcome()，用 publishConversationAssistantMessage() 把答案写进 conversation transcript，然后 completeRun()。

不进入 plan 不执行工具 Outcome passed=true

Task route

先 planTask() 生成 Task，再按 maxAttempts 循环执行 executeTask() 和 verifyTask()。verify passed 才发布 task_outcome；失败会重试直到次数耗尽。

默认 maxAttempts=2 toolResults 跨 attempt 累积 verifyTasks 默认 true

Thread route

executeThreadRoute() 把 routing decision 包成 parent task，然后通过 runThread 启动 child runAgentLoop({ kind: "thread" })。child 默认 verify:false，parent 再验证 thread output。

受 maxThreadDepth 限制 共享 emit 流 parent 记录 thread_output

事件、消息、Step 的三条轨道

AgentEvent 轨道

emit() 先把 event 推入 session.log 并更新 session.updatedAt，再调用外部 emit listener。

典型事件：session_created, chat_start, model_requested, task_start, tool_end, verification_end, outcome, error

messageLog 轨道

appendEventMessage() 只写本次 run 的 messageLog 并发 message_delta，用于 chat_start/chat_end 可观察 transcript。

phase_prompt、model_message、routing_decision、tool result、thread_output 默认在 execution scope。

session.messages 轨道

appendSessionMessage() 同时写 session.messages 和 messageLog。当前主要由 publishConversationAssistantMessage() 把 direct/task outcome 纳入 conversation transcript。

CLI 每轮进入前追加 user conversation message；CLI 是否打印由 formatOutcomeOutput() 和 console.log() 决定。

关键函数展开

collectTextAndStructured() - 模型事件收集器
  • 输入是 AsyncIterable<ModelStreamEvent>，每个 phase 都复用它消费 adapter/runtime 的 stream。
  • prompt_message 会变成 step entry prompt，同时发一个 phase_prompt execution message。
  • model_requested 会 consumeLimit("modelCalls")，然后发 AgentEvent model_requested。
  • text_delta 先缓冲；遇到 structured/tool/done/end 时 flush 成 assistant_text step 和 model_message。
  • structured_output 保存为当前 phase 的结构化结果；tool_call 会解析成 ToolCall 并发 tool_requested。
runPhase() - phase hook 包装器
  • 先调用 runtime.beforePhase(context, phase, input)，支持 abort、skip、替换 input。
  • 执行实际 runner，例如 routeRequest()、planTask()、executeTask()、verifyTask()。
  • 再调用 runtime.afterPhase(context, phase, output)，支持 abort、替换 output、或 retry。
  • retry 上限是 3 次，超过直接 throw。
executeToolCall() - 默认工具执行路径
  • 如果 context.config.runtime?.tools 存在，直接把工具调用交给 runtime port。
  • 默认路径先 findTool()，找不到就返回 unknown tool 错误并发 tool_end。
  • 找到工具后用 Schema.Compile(tool.parameters).Parse(args) 校验参数。
  • beforeToolCall 存在时发 approval requested/result；被拒绝时发 tool_blocked 并返回失败结果。
  • 允许后消耗 toolCalls limit，发 tool_start，调用 tool.execute()，再可经 afterToolCall review，最终发 tool_end。
executeThreadRoute() - 线程分支
  • 先检查 threadDepth >= maxThreadDepth，达到上限时直接失败，不创建子线程。
  • 把 routing decision 里的 prompt/task/goal 规范成 parent task，并发 task_created 和 task_start。
  • 调用 runThread()，默认实现来自 createNestedRunThread()，也就是再次调用 runAgentLoop({ kind:"thread" })。
  • child run 使用同一 model、stream、signal、limits、hooks 和 emit；child 自身默认 verify:false。
  • parent 收到 child outcome 后生成 ThreadTaskOutput，写入 execution message，并用 verify phase 判断该 thread output 是否满足 parent task。
completeRun() - 统一收口
  • 先确保 chat_end 只发一次。
  • 再发 outcome event。
  • 调用 createAgentRunResult()，普通 session 返回 { kind:"session", session, outcome, limitUsage, depth }。
  • thread result 额外带 parentSessionId、prompt、task、goal，并触发 thread_end。

当前实现里的重要边界

Agent 不是持久化层

它只保存运行态 state 和当前 session；CLI 在 promptWithLog() 里收集 steps，在 saveRunArtifacts() 里保存 session/steps。

loop.ts 仍然拥有 effect ordering

事件发射、message_delta、task attempts、verification、limit_exceeded、thread_end 和 outcome 顺序都在 runAgentLoop() 及其 helper 中维护。

Adapter 现在仍输出 generic stream events

OpenAI-compatible adapter 把 provider JSON 归一化后通过 structured_output / tool_call / text_delta 交给 loop；v0.4.3 计划中才要把 shared phase output contract 往 protocol 收拢。

Runtime hook 是高级插槽

AgentRuntimePort 能控制 phase before/after 和 tool runner，但不拥有 route/plan/execute/verify 的全局顺序，也不生成 final outcome。
